# 研究异常日志、超时降级与失败续跑

## 背景

调研过程中出现 `[ERROR] 系统错误，请稍后重试`，前端只能看到通用错误，后端日志也没有真实异常堆栈。

排查失败会话后确认：任务已进入 `SupervisorAgent` 的并行研究阶段，某个研究分支异常会冒泡到 `AgentPipeline._run_now()` 顶层 `except`，导致整个研究会话被标记为 `FAILED`。同时其他并行分支可能仍继续写入事件，造成“已失败但后续分支还在输出”的状态污染。

## 根因

1. `AgentPipeline._run_now()` 捕获所有异常后只发布通用 `ERROR` 事件，没有 `logger.exception(...)`，也没有任何开发态异常摘要。
2. `SupervisorAgent._execute_research_tasks()` 使用 `asyncio.gather(...)`，单个分支异常会直接打断整个 gather。
3. 分支级失败没有转换为可汇总的 `ResearchResult`，导致其他已完成或可继续完成的研究结果无法进入最终汇总。

## 修复方案

### 顶层异常日志

- 在 `backend-python/app/application/pipeline.py` 增加模块 logger。
- `AgentPipeline._run_now()` 的 `except Exception as exc` 改为：
  - 使用 `logger.exception(...)` 记录完整堆栈。
  - 保留原有 `FAILED` 状态更新和通用前端错误文案。
  - 当 `RESEARCH_OBSERVABILITY_CAPTURE_IO=true` 时，将脱敏后的异常摘要写入 `ERROR` 事件 `content`，便于开发环境直接看到错误类型和摘要。

### 并行研究分支兜底

- 在 `backend-python/app/application/agents.py` 增加模块 logger。
- `SupervisorAgent._execute_research_tasks()` 中每个 `run_task()` 内部捕获异常。
- 分支失败时：
  - 使用 `logger.exception(...)` 记录完整堆栈。
  - 发布 `研究分支失败: {title}` 的 `ERROR` 事件。
  - 将失败信息转换为 `ResearchResult`，让 Supervisor 继续汇总其他成功分支。

## 修改点总览

本轮围绕“错误可见、失败不拖垮整体、超时有兜底、失败后可继续”做了以下改动。

### 1. 顶层异常日志补全

- `AgentPipeline._run_now()` 顶层 `except Exception as exc` 增加 `logger.exception(...)`。
- 保留生产环境通用错误文案，避免把内部堆栈直接暴露给用户。
- 开发环境打开 `RESEARCH_OBSERVABILITY_CAPTURE_IO=true` 时，会把脱敏后的异常摘要写入 `ERROR` 事件 `content`，避免下次只能看到“系统错误，请稍后重试”。

### 2. 并行研究分支失败隔离

- `SupervisorAgent._execute_research_tasks()` 中每个分支单独捕获异常。
- 单个分支失败时记录后端日志，并发布 `研究分支失败: {title}` 事件。
- 失败分支会转换成一个 `ResearchResult` 继续参与汇总，其他成功分支不再被直接打断。

### 3. 模型连接重试

- `AgentScopeChatClient` 使用的 `OpenAIChatModel` 从 `max_retries=0` 调整为 `max_retries=2`。
- 对 `APIConnectionError: Connection error.` 这类瞬时模型网关/网络断连增加重试，降低单次抖动导致整条研究失败的概率。

### 4. 研究材料压缩超时降级

- `ResearcherAgent._compress_research()` 捕获压缩模型异常。
- 压缩失败时发布 `研究材料压缩失败，使用原始材料` 事件。
- 使用原始搜索材料、工具调用记录和研究主题生成可汇总的降级材料，避免该分支完全丢失。

### 5. 最终报告生成超时降级

- `ReportAgent.run()` 捕获最终报告模型异常。
- 报告模型失败时发布 `报告生成模型失败，使用兜底报告` 事件。
- 生成明确标记的 `研究报告（降级生成）`，把已收集的研究材料返回给用户，避免报告阶段超时后整条任务失败且无结果。

### 6. 失败后从失败处继续

- 顶层异常处理在标记 `FAILED` 前保存 `DeepResearchState` checkpoint。
- checkpoint TTL 从 30 分钟延长到 24 小时。
- 继续时如果已完成研究分支并进入报告阶段，会跳过 Scope 和已完成研究，直接重试报告生成。
- 如果 Redis checkpoint 过期，会从历史 `workflow_event` 尽量恢复研究简报、已完成分支、失败分支和报告阶段状态。
- 前端在失败状态增加“从失败处继续”按钮，点击后发送固定内容 `继续` 并重新连接 SSE。

## 行为变化

- 生产/默认环境：前端仍显示“系统错误，请稍后重试”，避免泄露内部异常。
- 开发环境：设置 `RESEARCH_OBSERVABILITY_CAPTURE_IO=true` 后，`ERROR` 事件会包含脱敏后的异常摘要。
- 单个研究分支失败不再直接拖垮整个研究任务；失败分支会作为一段失败结果进入后续汇总。
- 模型网关偶发 `APIConnectionError: Connection error.` 时，会在 AgentScope 模型层重试 2 次再判定失败。

## 追加排查记录

堆栈位置为 `ScopeAgent._clarify_user_instructions()` 调用 `AgentScopeChatClient.run_agent()`，再进入 `OpenAIChatModel` 的 Chat Completions 流式请求。此时 `total_input_tokens=0`、`total_output_tokens=0`，说明请求未成功返回模型 token。

对照验证：

- `192.168.31.36:8080` 端口可连，`/v1/models` 未带 key 时返回认证错误，说明服务端口在线。
- 同一模型配置的小流式 OpenAI SDK 调用连续成功。
- 同一 Scope 提示通过 AgentScope 直接调用可成功。

结论：这次不是模型配置错误，也不是并行研究分支异常；是模型网关或网络的瞬时连接失败。原代码 `max_retries=0` 会把一次连接失败直接放大为整条研究失败。

## TimeoutError 追加修复

继续执行同一个 RAG 研究任务时，后续又出现：

```text
TimeoutError:
```

这次失败点不在 Scope 阶段，也不是 MySQL/Redis 或服务存活问题。日志显示：

- 多个 `ResearcherAgent` 分支在 `_compress_research()` 调用模型整理研究材料时，流式读取等待超过 `LLM_TIMEOUT=300` 秒，被 `asyncio.wait_for(...)` 取消。
- 分支失败已经被转换为 `ResearchResult`，因此 Supervisor 可以继续汇总其他分支。
- 最后 `ReportAgent.run()` 在最终报告生成模型调用中再次等待超过 300 秒，异常冒泡到 `AgentPipeline._run_now()`，导致整个研究被标记为 `FAILED`。

修复：

- `ResearcherAgent._compress_research()` 增加降级：压缩模型失败时记录 `logger.exception(...)` 和 `ERROR` 事件，并用原始搜索材料/工具调用记录生成可汇总的分支材料。
- `ReportAgent.run()` 增加降级：最终报告模型失败时记录 `logger.exception(...)` 和 `ERROR` 事件，并生成“研究报告（降级生成）”，把已收集材料返回给用户，不再让整条任务因为报告成稿超时而失败。
- 新增单元测试覆盖压缩超时兜底和报告超时兜底。

追加验证结果：

- `compileall` 通过。
- `pytest -q` 结果为 `9 passed, 2 skipped, 1 warning`。

## 失败后从失败处继续

新增失败恢复能力：

- 失败时在顶层异常处理里先保存失败前 `DeepResearchState` 到 Redis checkpoint，再把会话标记为 `FAILED`。
- checkpoint 保留时间从 30 分钟延长到 24 小时，避免用户稍后回来点击继续时恢复点已经丢失。
- 如果失败前已经进入 `IN_REPORT`，或处于 `IN_RESEARCH` 但已经有 `supervisor_notes`，再次发送“继续”时会跳过 Scope 和已完成研究分支，直接重试报告阶段。
- 如果 checkpoint 已过期，则从历史 `workflow_event` 尽量恢复：
  - `SCOPE` 中的研究简报；
  - 已完成的 `RESEARCH` 分支材料；
  - `研究分支失败` 的错误信息；
  - 已进入 `REPORT` 阶段的状态。
- 前端在 `FAILED` 状态下显示“从失败处继续”按钮，点击后发送固定内容 `继续`，并重新连接 SSE。

追加影响范围：

| 文件 | 改动 |
|------|------|
| `backend-python/app/application/pipeline.py` | 失败前保存可恢复 checkpoint；恢复到已有研究材料时直接重试报告 |
| `backend-python/app/application/services.py` | 失败恢复状态构建与历史事件推断 |
| `backend-python/app/infrastructure/cache.py` | checkpoint TTL 延长到 24 小时 |
| `frontend/src/App.tsx` | FAILED 状态增加“从失败处继续”按钮 |
| `backend-python/tests/test_contracts.py` | 新增 checkpoint 恢复和历史事件恢复测试 |

## 质量影响

- 正常路径没有刻意降低质量，仍会优先使用模型压缩研究材料并由 `ReportAgent` 生成正式报告。
- 降级路径会影响报告质量。它的目标是“保结果不丢”，不是替代正式报告。
- 如果失败发生在最终报告阶段，从失败处继续会复用已完成研究材料重新生成报告，质量接近正常完整流程。
- 如果失败发生在研究分支或材料压缩阶段，从失败处继续会保留成功分支并带上失败/降级材料，最终报告可能出现部分主题覆盖不足、引用整理不完整或材料更粗糙。
- 后续如果追求更高质量，建议增加“补跑失败分支并重新生成正式报告”的入口，而不是只依赖兜底报告。

## 注意事项

- 开发态错误摘要复用现有 `RESEARCH_OBSERVABILITY_CAPTURE_IO` 开关，避免新增配置项。

## 研究材料压缩频繁失败的补充诊断

2026-07-01 继续排查“研究材料总是压缩失败”问题。后端日志显示，最近多次失败不是 Redis、MySQL、HTTP 服务或解析异常，而是 `ResearcherAgent._compress_research()` 调用模型流式输出时超过 `LLM_TIMEOUT=300` 秒，最终由 `asyncio.wait_for(...)` 抛出 `TimeoutError`。

关键证据：

- `/tmp/deep-research-backend.err` 中存在多次 `research compression failed ... TimeoutError`。
- 失败堆栈集中在 `backend-python/app/application/agents.py` 的 `_compress_research()`。
- `backend-python/app/infrastructure/llm.py` 中单次模型输出上限为 `max_tokens=16384`，默认模型调用超时为 300 秒。
- 最近两个 HIGH 任务累计 token 量已经达到约 34-36 万 input tokens、7 万 output tokens，说明研究分支材料规模很大。
- 当前压缩 prompt 要求“所有搜索结果、所有网页内容和摘要、所有事实、所有来源 URL”都保留，并要求“不要总结或改写”“不要丢失任何细节”。这会把“压缩”变成一次大型无损重排，实际成本接近再次生成一份长报告。

结论：

- 已实现的降级兜底不是根因，它只是避免压缩失败后整条任务直接失败。
- 根因是压缩阶段策略过重：输入材料太大，prompt 又要求近似无损保留，导致模型需要长时间读取、整理和输出，300 秒超时成为高概率事件。
- 单纯把 `LLM_TIMEOUT` 从 300 秒提高到 600 秒只能延长等待时间，不能解决材料规模和 prompt 目标不匹配的问题。

后续真正修复方向：

1. 将压缩 prompt 从“逐字保留全部”改为“预算内证据提取”。
2. 进模型前先做确定性裁剪：按 URL 去重、去掉 `thinkTool`/规划类内容、限制每个来源字符数、限制每个研究分支总字符数。
3. 将每个分支压缩输出限制到固定预算，例如 3000-6000 中文字。
4. 输出结构化证据包，而不是长篇原始材料搬运。建议字段包括：来源标题、URL、关键结论、关键数字、关键引用片段、适用范围、风险备注。
5. 最终报告阶段只消费结构化证据包，避免继续吞入原始网页材料。
6. 保留当前降级兜底和失败续跑能力，但不应让降级成为常态路径。

本次补充不改代码，只记录问题成因和下一阶段修复方向。
