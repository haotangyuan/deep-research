# 交互流程实时刷新修复

## 背景

提交方向修改后，页面需要手动刷新才会出现下一版研究方案，无法直接在当前对话中实时输出。

排查后确认：这不是单一按钮问题，而是所有“用户提交后继续工作流”的交互都存在同类风险。

## 根因

前端原流程在提交操作后直接重连 SSE：

1. 用户点击确认、提交修改、提交澄清或取消。
2. 后端接口在返回前后可能已经写入新的消息或事件。
3. 前端用 `resetCursor` 重连 SSE。
4. 后端 SSE 只会从 `Last-Event-ID` 之后回放；没有游标时不会自动回放历史。
5. 已经写入但错过推送窗口的消息只能在刷新页面时通过 `/messages` 全量加载出现。

因此刷新能看到内容，实时页面看不到。

## 影响流程

- HITL 方向确认：`APPROVE`
- HITL 修改方向：`REVISE`
- 交互式澄清表单提交
- 普通澄清文本提交
- 取消研究后的取消消息展示
- 普通继续发送消息的极快返回场景

## 修复方案

### 前端

- 新增统一快照同步逻辑 `syncResearchSnapshot()`：
  - 调用 `/api/v1/research/{id}/messages`
  - 同步 `status`、`messages`、`events`、token、时间字段
  - 同步历史侧栏状态
  - 根据最新 `sequenceNo` 更新 `Last-Event-ID`
- 提交类操作改为：
  1. 提交 API
  2. 拉取 `/messages` 权威快照
  3. 若状态仍是运行中，再从快照后的游标继续连接 SSE
- SSE 收到单条 `event` / `message` 时仍即时展示，同时触发快照同步，避免状态和交互卡片滞后。
- SSE 收到 `[DONE]` 后停止自动重连，并拉取最终快照。
- 消息和事件按 `id` / `sequenceNo` 合并，避免并发快照返回导致重复或倒退。
- 仅运行中状态保持 live SSE：
  - `QUEUE`
  - `START`
  - `RUNNING`
  - `IN_SCOPE`
  - `IN_RESEARCH`
  - `IN_REPORT`

等待用户输入的状态不再无意义重连 SSE：

- `NEED_CLARIFICATION`
- `AWAITING_DIRECTION_CONFIRM`

### 后端

- `/messages` 响应补充会话元信息：
  - `title`
  - `modelId`
  - `budget`
- 取消接口改为通过 `event_publisher` 发布取消事件和取消消息，确保前端可实时收到。

## 影响范围

| 文件 | 改动 |
|------|------|
| `frontend/src/App.tsx` | 新增快照同步、SSE 游标维护、消息/事件合并、提交后同步流程 |
| `frontend/src/services/api.ts` | `ResearchMessageResponse` 补充 `title`、`modelId`、`budget` |
| `backend-python/app/domain/dto.py` | `ResearchMessageResp` 补充会话元信息 |
| `backend-python/app/application/services.py` | `/messages` 返回会话元信息；取消流程改为实时发布 |
