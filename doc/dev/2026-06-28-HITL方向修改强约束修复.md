# HITL 方向修改强约束修复

## 背景

用户在研究方向确认环节选择 `REVISE` 并提交修改意见后，后端会保存 `state.hitl_feedback`，随后重新进入 `ScopeAgent` 生成新版研究简报。

问题表现为：用户明确要求缩小或改写时间范围时，模型仍可能沿用旧研究简报或历史消息中的范围。例如用户要求改为指定年份，模型在新版简报中弱化成“近2-3年”，导致后续研究仍按错误时间范围执行。

## 根因

`confirm_direction(REVISE)` 已经把修改意见写入了状态：

- 保存用户消息：`修改意见: ...`
- 写入 `state.hitl_feedback`
- 清空旧 `state.research_brief`
- 重新进入 Scope 阶段

但 `ScopeAgent` 生成研究简报时，只把完整聊天历史作为普通上下文传给 `TRANSFORM_MESSAGES_INTO_RESEARCH_TOPIC_PROMPT`。修改意见虽然存在于历史消息里，但没有被明确声明为高优先级约束，也没有要求它覆盖旧研究简报或旧确认内容。

因此 LLM 会把用户修改意见当成普通历史消息重新总结，容易被旧范围、旧 brief 或模型自身的“合理时间范围建议”稀释。

## 修复方案

### 后端

- 在 `ScopeAgent` 中新增 `_hitl_feedback_section()`。
- 当 `state.hitl_feedback` 存在时，将其渲染为：
  - `<HumanRevision priority="highest">`
  - 明确说明这是方向确认环节的人工修改意见
  - 明确要求覆盖历史消息、旧研究简报或旧确认消息中的冲突内容
  - 明确要求时间范围必须严格使用用户指定范围，不得扩展、近似或改写为相对时间表达
- 在 `ScopeAgent._write_research_brief()` 中，把 `HumanRevision` 注入研究简报生成 prompt。
- 在 `ScopeAgent._clarify_user_instructions()` 中也注入同一约束，避免 revise 后澄清判断阶段继续被旧上下文影响。
- 在 `TRANSFORM_MESSAGES_INTO_RESEARCH_TOPIC_PROMPT` 中新增“人工修改意见优先”原则：
  - `HumanRevision` 是最高优先级约束
  - 与历史消息、旧研究简报或旧确认消息冲突时，以 `HumanRevision` 为准
  - 指定时间范围后必须原样落实，不得扩展为“近几年”“近2-3年”“最近”等

### 兼容性

- 不修改 REST API 请求/响应结构。
- 不修改 SSE 事件结构。
- 不修改 MySQL / Redis 持久化结构。
- 保留已有 `hitl_feedback` 状态字段和队列恢复逻辑。

## 影响范围

| 文件 | 改动 |
|------|------|
| `backend-python/app/application/agents.py` | ScopeAgent 注入 `HumanRevision` 强约束；生成 brief 和澄清判断都使用该区块 |
| `backend-python/app/application/prompts.py` | 研究简报 prompt 新增人工修改意见优先规则；Context 支持 `hitl_feedback_section` |
| `backend-python/tests/test_contracts.py` | 新增单测，捕获 ScopeAgent prompt，验证 HITL 修改意见和时间范围硬约束已进入模型输入 |
| `backend-python/tests/debug_scope_once.py` | 调试脚本补充空 `hitl_feedback_section`，避免模板变量缺失 |

## 验证

```bash
cd backend-python
conda run -n deep-research-py python -m compileall -q app tests
conda run -n deep-research-py pytest -q tests/test_contracts.py
```

结果：

- Python 编译通过
- `tests/test_contracts.py` 通过
- 本地 MySQL/Redis 依赖的测试按原有逻辑跳过：`6 passed, 2 skipped`

新增关键测试：

- `test_scope_agent_injects_hitl_revision_as_hard_prompt_constraint`

测试断言包括：

- prompt 中包含 `<HumanRevision priority="highest">`
- prompt 中包含用户原始修改意见
- prompt 中包含“覆盖历史消息、旧研究简报或旧确认消息”
- prompt 中包含“不得扩展、近似或改写”
- prompt 中明确禁止“近2-3年”等相对时间弱化

## 操作建议

如果某个会话当前停在 `AWAITING_DIRECTION_CONFIRM`，且页面上的研究方向仍是旧范围，不要点击 `APPROVE`。

应在后端修复部署后重新提交 `REVISE`，把正确时间范围作为修改意见再次发送。新版 ScopeAgent 会将该修改意见作为最高优先级约束重新生成研究简报，确认无误后再进入最终研究阶段。

## 后续观察点

- 观察新版研究简报是否逐字保留用户指定的时间范围。
- 观察 SupervisorAgent 拆解任务时是否继续继承新版 brief 中的时间范围。
- 若后续研究阶段仍出现时间范围漂移，可在 SupervisorAgent / ResearcherAgent prompt 中继续传递结构化约束，但本次根因位于 ScopeAgent brief 生成阶段。
