# 2026-06-26 HITL 功能添加

## 背景

当前深度研究系统是纯自动化的三阶段流水线：ScopeAgent → SupervisorAgent(并行搜索) → ReportAgent，中间只有 `NEED_CLARIFICATION`（AI 觉得问题不清楚时被动反问）这一个停顿点。用户需要在 AI 开始大规模并行搜索、消耗大量 token 之前，有机会确认和修正研究方向。

## 方案

### 架构决策

在 **ScopeAgent 产出 ResearchBrief 之后、SupervisorAgent 启动之前** 插入 HITL 检查点。用户可以对研究方向做 APPROVE（确认继续）或 REVISE（反馈修改意见后重新分析）。

**为什么不在搜索前或工具调用前做 HITL？** 一次深度研究可能触发几十次搜索/工具调用，每次都打断用户不可接受。正确做法是在搜索开始前（ScopeAgent 之后）让用户确认大方向，方向对了所有搜索才有意义。

### 1. HITL 触发与状态暂停

新增 `WorkflowStatus.AWAITING_DIRECTION_CONFIRM` 状态。`AgentPipeline.run()` 在 Phase 1（ScopeAgent）成功后检查 `state.getHitlMode()`：

```java
if ("DIRECTION_ONLY".equals(state.getHitlMode())) {
    checkpointStore.save(researchId, state);  // Redis 持久化
    state.setStatus(WorkflowStatus.AWAITING_DIRECTION_CONFIRM);
    updateResearchSession(researchId, AWAITING_DIRECTION_CONFIRM, state);
    eventPublisher.publishEvent(researchId, DIRECTION_CONFIRM, ...);
    eventPublisher.publishMessage(researchId, "assistant", "...");
    return;  // 暂停流水线，等待用户操作
}
```

恢复时 `state.isSkipScopePhase() == true` 直接进入 Phase 2。

### 2. Redis Checkpoint 持久化

新建 `CheckpointStore.java`，将整个 `DeepResearchState` 序列化到 Redis：

- Key: `research:{id}:checkpoint`
- TTL: 30 分钟
- 序列化方式: Jackson JSON（`DeepResearchState` 已有 `@Data` + `@Builder`，天然可序列化）
- 恢复：反序列化后直接传入 `agentPipeline.run()`，`skipScopePhase` 标记跳过 ScopeAgent

对比"DB 列持久化"方案优势：无需加表加列，完整保存所有状态，未来加新 HITL 检查点零 schema 变更。

### 3. 确认/修改 API

```
POST /api/v1/research/{researchId}/direction-action
Body: { action: "APPROVE" | "REVISE", feedback?: string }
```

- **APPROVE**：`skipScopePhase = true`，从 Redis 恢复 state，直接执行 SupervisorAgent
- **REVISE**：`skipScopePhase = false`，清空 `researchBrief`，ScopeAgent 重新运行

使用 CAS 更新保证幂等：`WHERE status = 'AWAITING_DIRECTION_CONFIRM'`

### 4. 暂停/取消研究中研究

新增 `POST /api/v1/research/{researchId}/cancel` 端点，设置状态为 `CANCELLED`。Pipeline 在每个阶段间隙检查 DB 状态，发现 CANCELLED 后立刻停止执行。

- `AgentPipeline.executePhase2And3()` 阶段间检查
- `SupervisorAgent.run()` 执行完成后、发布"准备生成报告"事件前检查
- `cancelledRef` 前端 ref 屏蔽取消后残余 SSE 事件

### 5. 前端交互

HITL 确认按钮直接嵌入在 AI 回复的消息气泡内，不独立弹窗。用户输入框在研究过程中始终可见（不再根据状态隐藏）。终端状态（COMPLETED/FAILED/CANCELLED）下发消息自动跳转到 `/new` 新建研究。

## 验证

1. **方向确认流程**：发起研究 → ScopeAgent 产出 ResearchBrief → 状态变为 `AWAITING_DIRECTION_CONFIRM` → AI 消息气泡中出现"确认方向，开始研究"按钮 + 修改输入框 → 点确认 → SupervisorAgent 开始执行 → 完成报告
2. **修改方向流程**：方向确认态 → 输入修改意见并提交 → ScopeAgent 重新运行 → 再次到确认态 → 确认 → 完成
3. **关闭 HITL**：`hitlMode = "NONE"` → 研究直接执行不停顿
4. **取消研究**：研究中点「取消」按钮 → 状态变为 CANCELLED → Pipeline 在下一阶段检查点停止 → 残余 SSE 事件被前端屏蔽
5. **取消后发新消息**：已取消页面输入问题 → 自动跳转到 `/new` → 创建新研究正常执行
6. **多模型 Arena**：各自独立 researchId，HITL 互不影响
7. **Prompt 日期**：所有 Agent 不再反问"您说的今年指哪一年"

## 遗留问题

- 取消操作**异步执行完成后**才停止 Pipeline（当前阶段正在执行的 LLM 调用无法中断），前端通过 `cancelledRef` 屏蔽了残余事件，用户不会感知到 Pipeline 的后台延迟。
- 当前仅支持 `DIRECTION_ONLY` 级别 HITL（ScopeAgent 后暂停），需更多 HITL 检查点（如任务拆解确认）只需在对应位置加类似暂停逻辑，再利用 Redis Checkpoint 恢复即可。
