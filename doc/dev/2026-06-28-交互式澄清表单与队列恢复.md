# 交互式澄清表单 & 队列恢复

## 背景

1. **澄清表单**：当 AI 需要向用户澄清信息时，原来是纯文本问答，体验差。改为结构化交互表单（多选题复选框 + "其他"填写框 + 问答题文本框 + 提交按钮）。
2. **队列恢复**：研究任务队列是内存级 `asyncio.Queue`，后端重启后 QUEUE 状态的任务永远卡住，无法自动恢复。

## 方案

### 交互式澄清表单

**后端**：
- 新增 `CLARIFY_FORM` 事件类型（`constants.py`）
- 更新 `CLARIFY_WITH_USER_INSTRUCTIONS` prompt，LLM 可输出结构化 `clarificationForm` JSON（含 `title`、`questions[]`，支持 `multi_choice` + `allowOther`、`open_ended` 两种题型）
- `WorkflowEventDTO` 新增 `form_data` 字段
- `cache.py` 中 `save_event()` 和 `load_from_db()` 自动解析 `CLARIFY_FORM` 事件的 `content` JSON 为 `form_data`
- `ScopeAgent._clarify_user_instructions()` 在 NEED_CLARIFICATION 时，若 LLM 返回了 `clarificationForm`，额外发布 `CLARIFY_FORM` 事件

**前端**：
- `api.ts` 新增 `ClarificationQuestion`、`ClarificationForm` 接口，`WorkflowEvent` 新增 `formData` 字段
- `App.tsx` 新增澄清表单逻辑：
  - 检测 `NEED_CLARIFICATION` 状态 + `CLARIFY_FORM` 事件
  - 多选题渲染为复选框，选项为"其他"时始终显示填写框
  - 问答题渲染为 textarea
  - 右下角黑色提交按钮
  - 提交时编译答案为格式化文本，走现有 `sendMessage` API
  - 向后兼容：LLM 不输出 `clarificationForm` 时仍用纯文本提问

### 启动时队列恢复

- `pipeline.py` 中 `ResearchTaskQueue.start()` 完成后调用 `_recover_interrupted_tasks()`
- 启动恢复策略：
  - `QUEUE`：重建 `DeepResearchState`，加载模型、聊天历史、预算，重新放入内存队列
  - `START` / `IN_SCOPE` / `IN_RESEARCH` / `IN_REPORT`：标记为 `FAILED`，写入“服务重启导致任务中断”事件，避免重复执行半截任务
  - `NEED_CLARIFICATION` / `AWAITING_DIRECTION_CONFIRM`：等待用户输入，不自动恢复执行
  - `COMPLETED` / `FAILED` / `CANCELLED`：终态，不处理
- 恢复 `QUEUE` 时：
  - 若 Redis checkpoint 仍存在且最近用户消息是“确认研究方向，开始执行研究”，从 checkpoint 恢复并跳过 Scope，直接进入研究阶段
  - 若最近用户消息是“修改意见: ...”或“请重新调整研究方向”，按数据库聊天历史重建 state，重新进入 Scope
  - 若没有 checkpoint，则按数据库聊天历史重建 state
- 恢复失败（模型缺失、队列容量不足等）时，将任务标记为 `FAILED` 并写入错误事件

- 前端 TypeScript 编译通过

## 遗留问题

- `clarificationForm` 是否生成取决于 LLM，无法保证每次都输出结构化表单（纯文本提问作为 fallback）
- 队列恢复依赖数据库中的聊天历史和模型信息完整，缺失时该条标记为 `FAILED`
- 执行中状态目前不会从中间阶段继续执行，而是保守标记为 `FAILED`，后续可在引入阶段级 checkpoint 后增强为断点续跑
