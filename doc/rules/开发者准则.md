# 开发者准则

本项目为小团队协作，以下规范帮助保持代码质量、降低协作成本。

## 1. 分支管理

- **每次开发必须在独立分支上进行**，分支命名：`feature/xxx`、`fix/xxx`、`refactor/xxx`
- 开发完成并自测通过后，通过 PR 合入 `main` 分支
- 禁止直接向 `main` 推送未经测试的代码
- PR 描述应包含：改动内容、影响范围、测试方式

## 2. 文档同步更新

每次开发完成后，**必须同步更新以下文档**：

- **`CLAUDE.md`**：如果架构、构建方式、关键类有变化，及时更新，确保 Claude Code 能正确理解项目
- **`README.md`**：如果新增功能、修改配置项、改变架构，更新对应章节
- **`doc/dev/`**：编写本次开发记录（见第 3 条）

## 3. 开发记录

每次开发完成后，在 `doc/dev/` 下新建文档，命名格式：`YYYY-MM-DD-变更内容.md`

内容应包含：
- **背景**：为什么要做这个改动
- **方案**：怎么做的，关键设计决策
- **影响范围**：改了哪些文件，是否涉及数据库变更、配置变更
- **测试**：怎么验证的，结果如何
- **遗留问题**：如有未解决的问题或后续优化点

## 4. 提交规范

Commit message 使用中文，格式：

```
<类型>(<模块>): <简要描述>

<可选的详细说明>
```

类型：`feat`、`fix`、`refactor`、`docs`、`test`、`chore`

示例：
```
feat(agent): 新增 ResearcherAgent 并行执行能力
fix(sse): 修复断线重连后事件丢失问题
refactor(runtime): 拆分 AgentscopeJavaChatClient 为多个职责类
docs(dev): 添加可观测链路开发记录
```

## 5. 代码风格

- 使用 Lombok 注解（`@Slf4j`、`@RequiredArgsConstructor`、`@Data`），不要手写 getter/setter/log
- 中文注释，关键逻辑必须有注释
- 方法不超过 50 行，超过则拆分
- 不提交 `.env` 文件，敏感信息通过环境变量注入
- 新增配置项必须在 `.env.example` 中添加说明

## 6. 数据库变更

- 新增表或字段，必须同步更新 `src/main/resources/data.sql`
- 不使用 ORM 迁移工具，schema 变更通过 SQL 文件管理
- 变更前在 `doc/dev/` 记录变更内容和影响

## 7. 测试要求

- 修改 Agent 逻辑后，至少做一次完整的端到端闭环测试（发起研究 → 完成 → 检查结果）
- 修改可观测链路后，在 Langfuse 中验证 span 层级是否正确
- 修改 API 接口后，用 curl 或 Scalar UI 验证请求/响应
- 有条件时编写单元测试，特别是工具注册、消息转换、状态管理等核心逻辑

## 8. 依赖管理

- 新增依赖前先评估必要性，避免引入过重的库
- 版本号在 `pom.xml` 的 `<properties>` 中统一管理
- 升级依赖后必须跑通完整流程测试

## 9. 协作沟通

- 开发前在群里同步要做的事情，避免重复劳动
- 遇到阻塞问题及时沟通，不要闷头卡太久
- PR review 至少一人通过后才能合入
- 重大架构变更需要团队讨论后再动手

## 10. 环境一致性

- 本地开发使用 `.env` 文件管理环境变量，参考 `.env.example`
- 确保 MySQL、Redis 版本与生产环境一致
- Java 版本锁定为 21，不要使用更高版本的语法特性
