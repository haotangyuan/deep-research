## 1. 项目定位与概述

Deep Research 是一个基于多智能体（Multi-Agent）协作的**自动化深度研究平台**。核心流程：

```
用户输入研究主题 → ScopeAgent 澄清需求 → SupervisorAgent 拆解任务 →
    ResearcherAgent 并行深度搜索 → ReportAgent 生成 Markdown 报告
```

项目参考 LangChain `open-deep-research` 思路，采用 **Spring Boot 3 + Java 21** 后端 + **React 18 + Vite** 前端架构，通过 **SSE（Server-Sent Events）** 实现实时进度推送。

**核心特点**：
- 可切换 Agent 运行时（AgentScope Java / LangChain4j）
- 三级预算控制（MEDIUM/HIGH/ULTRA）防止成本失控
- 双层缓存策略（Tavily 搜索缓存 + 网页摘要缓存）
- OpenTelemetry + Langfuse 可观测链路
- SSE 实时推送 + Redis 断线重连

## 3. 核心架构详解

### 3.1 Agent Pipeline 三阶段流水线

```
AgentPipeline.run()
  ├── Phase 1: ScopeAgent
  │     └── 分析用户需求，判断是否需要澄清
  │          ├── 需要澄清 → NEED_CLARIFICATION 状态 + 向用户提问
  │          └── 无需澄清 → 生成 ResearchBrief
  │
  ├── Phase 2: SupervisorAgent
  │     ├── planResearchTasks() → 拆解为 ResearchTask[]
  │     ├── executeResearchTasks() → 并发执行（maxConcurrentUnits 控制）
  │     │     └── ResearcherAgent × N（每路独立 DeepResearchState 分支）
  │     │           └── ResearcherAgent.plan() → 多轮迭代（最多 maxSearchCount×2）
  │     │                 ├── tavilySearch → SearchAgent（搜索+摘要）
  │     │                 ├── thinkTool → 反思工具
  │     │                 └── 其他工具 → ToolRegistry 反射执行
  │     └── summarizeSupervisorResults() → 合并研究笔记
  │
  └── Phase 3: ReportAgent
        └── 整合所有笔记生成 Markdown 报告
```

### 3.2 关键设计模式

| 模式 | 实现 | 评估 |
|------|------|------|
| **策略模式** | `AgentRuntime` 双实现，`@ConditionalOnProperty` 切换 | ✅ 干净解耦 |
| **装饰器模式** | `ResearchObservation` 包装模型/工具调用，不侵入业务 | ✅ 可观测性设计良好 |
| **上下文传播** | `ResearchOtelContext` ThreadLocal 栈 + `withOtelContext` 跨线程 | ✅ 解决 Reactor 异步上下文丢失 |
| **双重缓存** | Tavily 搜索缓存 + Summary 缓存（TTL + LRU + in-flight 合并） | ✅ 性能优化到位 |
| **幂等设计** | CAS 更新队列状态 + SSE Last-Event-ID 重放 | ✅ 防止重复执行和断线丢失 |
| **反射工具执行** | `@ResearchTool` + `ReflectiveResearchToolExecutor` | ⚠️ 遍历 `getMethods()` 包含继承方法 |
| **上帝对象** | `DeepResearchState` 包含所有阶段数据 | ⚠️ 字段多，难以维护 |

### 3.3 工具系统

```
@ResearchTool（方法级）+ @ResearchToolParam（参数级）
         │
         ▼
  ToolRegistry（Spring 扫描 + 按阶段分组）
         │
         ├── supervisorTools → ResearchTask[] / thinkTool
         └── researcherTools → tavilySearch / thinkTool / researchComplete
         │
         ▼
  ReflectiveResearchToolExecutor（反射调用目标 Bean）
```

### 3.4 数据库设计（5 张核心表）

| 表名 | 用途 |
|------|------|
| `user` | 用户账号（用户名 + 密码，Google OAuth 预留） |
| `research_session` | 研究会话、状态机、模型绑定、预算、Token 统计 |
| `model` | AI 模型配置（全局/用户自定义，OpenAI-compatible） |
| `chat_message` | 用户/助手消息，含最终研究报告 |
| `workflow_event` | Agent 工作流事件，支持父子层级树 |

### 3.5 SSE 推送机制

```
Agent → EventPublisher → (持久化 DB + Redis ZSet) → SseHub → 浏览器
                                                    │
                                          lastEventId 断线重连
                                          Redis ZSet 回源 DB 补事件
```

- 双层 `ConcurrentHashMap<researchId, Map<clientId, SseEmitter>>`
- 30 秒心跳保活
- Redis ZSet 时序索引（score = sequenceNo）
- 支持 `Last-Event-ID` 断线续传

## 4. 前端架构

### 4.1 路由结构

```
/*          → AuthProvider 包裹所有路由
/research/:id → ResearchPage（已有研究）
/new          → ResearchPage（新建研究）
/arena        → ArenaPage（多模型对比）
/ (implicit)  → Navigate to /new
```

### 4.2 状态管理

- **最小化方案**：无 Redux/Zustand，仅 `AuthContext` 一个 React Context
- 页面级状态全部使用 `useState`/`useRef`/`useMemo`
- 跨组件同步：`window` 自定义事件（`refreshHistory`、`historyStatusUpdate`）

## 5. 代码质量评估

### 5.1 优势 ✅

1. **架构清晰**：Controller → Service → Pipeline → Agent → Tool 分层明确
2. **框架解耦**：AgentRuntime 策略模式实现 AgentScope/LangChain4j 无缝切换
3. **可观测性完备**：OTel 三层 Span 层级 + Langfuse + 自动脱敏
4. **性能优化到位**：双层缓存、并发控制、异步任务队列
5. **SSE 设计成熟**：断线重连、心跳保活、Redis 时序回源
6. **前端 UX 细致**：自动滚动、复制反馈、动画过渡、预算/模型选择器

### 5.2 待改进 ⚠️

1. **`DeepResearchState` 是上帝对象**：包含所有阶段数据，字段多难以维护，建议按阶段拆分为独立 DTO
2. **`ToolRegistry` 遍历 `getMethods()`**：包含继承方法（如 `toString`、`equals`），应改为 `getDeclaredMethods()` 或加白名单
3. **`App.tsx` 单文件 ~1310 行**：整个路由、侧边栏、研究页面都堆在一个文件里，建议拆分
4. **`ArenaPage` ~727 行**：同样偏大，可拆分为 `ArenaCard`、`ArenaGrid` 等子组件
5. **缺少单元测试**：`src/test/` 目录不存在，无测试覆盖
6. **密码存储**：明文存储，文档标注需改为 BCrypt
7. **数据库无迁移工具**：变更通过手动管理 `data.sql`
8. **文档不一致**：`maxConcurrentUnits` 在技术文档中标注为"无并发实现"，但代码中已实现
9. **`ModelHandler.chatClientPool` 生命周期依赖 `finally`**：若 pipeline 崩溃可能导致连接池泄漏
10. **重复实现**：`TavilyClient` 和 `SearchAgent` 各自实现了 `CompletableFuture` in-flight 合并逻辑

### 5.3 安全风险 🔴

| 风险 | 描述 | 建议 |
|------|------|------|
| 密码明文 | 用户密码未加密存储 | 改用 BCrypt |
| API Key 暴露 | `model.apiKey` 明文存储 | 考虑加密存储或密钥管理服务 |
| 无 CSRF 防护 | Spring Boot 默认 CSRF 仅拦截 POST，但 API 应完全禁用 | 确认 API 无 CSRF 风险 |
| JWT 密钥硬编码风险 | 建议从环境变量读取而非代码硬编码 | 检查配置 |

## 8. 总结与建议

### 总结

Deep Research 是一个**架构成熟、功能完整**的多智能体深度研究平台。核心技术链路（Agent Pipeline、工具系统、预算控制、SSE 推送、可观测性）均已实现，具备生产部署的基础能力。

### 优先改进建议

1. **🔴 高优先级**：添加单元测试（至少覆盖 Pipeline 和 Agent 核心逻辑）
2. **🔴 高优先级**：密码改用 BCrypt 加密存储
3. **🟡 中优先级**：拆分 `DeepResearchState` 为阶段独立 DTO
4. **🟡 中优先级**：拆分 `App.tsx` 和 `ArenaPage.tsx` 为独立组件文件
5. **🟢 低优先级**：引入 Flyway/Liquibase 数据库迁移管理
6. **🟢 低优先级**：统一 `TavilyClient` 和 `SearchAgent` 的 in-flight 合并逻辑
